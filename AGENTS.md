# Repository Guide

## 目的

このリポジトリは、複数のブラウザゲームを 1 つのロビーと共通ランキング基盤で運用するための親リポジトリです。

## 基本構成

- `games/`
  各ゲーム本体を置く
- `platform/`
  ロビー、API、ランキング保存、ゲーム一覧定義を置く
- `docs/`
  仕様書や運用メモを置く

## 変更ルール

- 新しいゲームは必ず `games/<slug>/` 配下に置く（原則として **TypeScript + Phaser** で実装すること）
- ロビー表示内容は `platform/games.mjs` を更新する
- ゲーム配信パスは `platform/server.mjs` を更新する
- replay 検証が必要なゲームは `platform/adapters/` に追加する
- 共通処理は各ゲーム側へ重複実装せず `platform/` に寄せる
- ロビーのカード背景用サムネイルは、実際のゲーム画面ではなく、そのゲームの内容にあった「レトロゲーム風・シンセウェイヴテイスト」のハイクオリティなイラストをAI生成し `platform/public/assets/thumbnails/<slug>.png` に設定する


## 命名ルール

- フォルダ名は小文字の kebab-case を使う
- ルート直下にゲーム本体や無関係なプロジェクトを増やさない（`games/` 配下、または別リポジトリで管理する）
- 空白を含む新規フォルダ名は作らない
- 開発中の一時ファイル（`*.bak`, `tmp*`等）をルートに残さない

## Git運用

- 統合サイト全体を親リポジトリ 1 つで管理する
- `games/` 配下に入れ子の `.git` を作らない
- 生成物、キャッシュ、ローカルDBは原則コミットしない
- **AIベンチマーク/シード**: 初期状態でランキングを稼働させるための AI 用 JSON データは例外的に `git add -f` 等で追跡対象に含める場合がある。リクエスト時に無視されていないか確認すること。
- **デプロイ**: Cloudflare Workers へは `npm run cf:deploy` (`wrangler`) で反映する。本番反映時はブランチとデプロイ状況を確認すること。

## README運用

- ルート `README.md` は全体の入口として保つ
- ゲーム固有の説明は各ゲーム配下の README に置く
- 大きな構成変更をしたら README と `docs/` を同時に更新する

## 追加時の最小チェック

1. `platform/games.mjs` の一覧に出るか
2. ロビーからゲームへ遷移できるか
3. 必要な静的ファイルが配信されるか
4. `.gitignore` に不要な生成物が漏れていないか
5. `platform/public/assets/thumbnails/<slug>.png` に、ゲームのイメージに合った「レトロゲーム風・シンセウェイヴテイスト」のハイクオリティなAI生成イラストを配置しているか


## AI 開発運用ルール

エンジニア（ユーザー）と AI の円滑な協力のために、以下のプロセスを遵守します。

### 1. 計画先行（Planning First）
AI は作業を開始する前に、必ず `implementation_plan.md` を作成（または更新）します。
計画には「どのファイルのどの箇所をどのように変更するか」を具体的に細かく記述します。

### 2. 承認プロセスの固定
- AI は計画を作成後、必ずユーザーに提示し「承認（LGTM等）」を求めます。
- **承認を得るまでコードの改変は開始しません。**
- ユーザーから承認が得られた後は、AI は作業完了まで自律的に進め、逐次の承認を必要としません。

### 3. 未説明事項の禁止（Strict Adherence）
AI は、**計画書に記載・説明していない変更を絶対に行いません。**
推測で範囲外の修正を加えることは厳禁とし、必要が生じた場合は再度計画を提示して承認を得るものとします。

## Antigravity 専用 開発運用ルール

AIアシスタント「Antigravity」が作業を行う場合は、上記に加えて以下の制限を厳守します。
（※このセクションは Antigravity 以外のAIモデルには適用されません）

### 1. 作業前の具体的説明と承諾
Antigravity は、ユーザーの指示に対して具体的な作業内容を説明し、承諾を得るまで実行に移りません。

### 2. 想定作業時間の試算
作業の計画提示（implementation_plan.md等）の際、必ず**「完了までの想定作業時間（分）」**の試算を含めます。ユーザーはこの試算を含めて承認を判断します。

### 3. 3倍超過時の強制中断
作業の進捗が**当初の想定時間の３倍を超過する見込みとなった場合、Antigravity は直ちに作業を中断**します。
その後、遅延の理由を具体的に説明し、再度ユーザーの承諾を得るまで作業を再開しません。
